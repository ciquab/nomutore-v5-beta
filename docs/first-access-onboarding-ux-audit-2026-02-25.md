# 初回アクセス導線・初回記録までのUX監査（2026-02-25 / フィードバック反映版）

## 目的
- 初回離脱を減らす
- 初回記録（1件目）までの到達率を高める
- 「自分が何を選択したか」が迷わず把握できる状態にする

## いただいた指摘への回答と方針修正

### 1) 「ビール記録モーダルは任意項目が既に折りたたみでは？」
- ご指摘のとおり、現在UIは既に任意項目を一定程度抑制できています。
- そのため「さらに減らす」は必須施策ではなく、**優先度を下げます**。
- 代わりに、初回改善の主軸は「導線（どこに連れていくか）」に寄せます。

**修正方針**
- フォーム削減は Phase 2 の検討項目へ後退。
- まずは初回記録地点への到達率改善（タブ誘導・CTA強化）を優先。

### 2) 「初回記録はビールとは限らない。Recordタブ誘導が良い」
- 完全同意です。
- 旧提案の「ビールモーダル自動起動」は偏りがあるため、**標準導線を Recordタブ自動遷移に変更**します。

**修正方針**
- オンボ完了直後は `Record` タブを開く。
- Recordタブのファーストビューに「ビールを記録」「運動を記録」を同格で提示。
- 必要なら初回のみミニガイド（1行）を表示し、押下先を明確化。

### 3) 「推奨値で仮開始→後で修正は手間。記録対象で分岐できないか」
- 非常に合理的です。**分岐オンボ**を採用候補に上げます。

**修正方針（分岐案）**
- Welcomeで「最初に記録したいのは？」を先に聞く。
  - `ビールを記録したい`
  - `運動を記録したい`
- 分岐仕様:
  - ビール開始: プロフィールは任意（スキップ可）。
  - 運動開始: 体重など計算に影響する最低限入力を要求。
- **プロフィール未設定ガード（追加）**:
  - ビール先行でプロフィールをスキップしたユーザーが運動記録をしようとした場合、運動記録前にプロフィール設定を必須化する。
  - 表示タイミング案は2つ（併用可）:
    1. 初回ビール記録完了直後に「運動記録の前にプロフィール設定が必要」アラートを表示。
    2. 運動記録モーダル起動時にガードダイアログを表示し、プロフィール設定へ遷移。
- これにより、入力負荷を「必要な人にだけ」発生させつつ、運動記録の精度を担保する。

### 4) 「チュートリアルの過不足（LiverRankはあるがCheckStatusはない等）」
- 現状ツアーは要素説明中心で、**情報の粒度が不均一**です。
- 初回離脱抑制の観点では、カード単位より「タブ単位の行動導線」重視が有効です。

**修正方針（チュートリアル再編）**
- 初回ツアーを2層に分離:
  1. **必須ツアー（3ステップ前後）**
     - Home（何を見るか）
     - Record（最初の1件を入れる場所）
     - Homeオーブで反映確認（補助で Cellar > Logs を案内）
  2. **任意ツアー（詳細）**
     - LiverRank / CheckStatus / 各カードの個別説明
- 必須ツアー内では、説明対象の偏りをなくし「まず使う」に必要な情報だけ残す。

### 補足) 「?ボタンの挙動は変えるのか？」への回答
- `?` ボタン自体の役割は変更しません（従来どおりヘルプモーダルを開く）。
- 変更点はヘルプモーダルの中身で、"詳細ツアー開始"を追加する分岐型にします。
- これにより、従来のテキストヘルプ利用者と、操作ガイドを見たい利用者の両方をカバーできます。

### 補足) 「詳細ツアーの内容は今より増やすのか？」への回答
- 基本方針は**量を増やすことではなく、構造を分離すること**です。
- 必須ツアーから移した既存説明（カード個別説明）を中心に再配置し、初回に不要な情報を任意側へ逃がします。
- したがって、初期リリース時点では詳細ツアーの総量は「大幅増」ではなく、
  - 既存説明の再編（主）
  - 説明粒度の均一化（LiverRank/CheckStatus などの偏り是正）
  を目的とします。
- 追加説明は、Phase 3 の計測で「理解不足が残る箇所」が確認された場合に限定して増やします。

### 5) 「未リリースなのでA/Bテストは難しい」
- ご指摘の通りです。A/B前提は現実的ではないため、**段階リリース + 計測**に置き換えます。

**修正方針**
- A/Bの記載は削除。
- 代替として:
  - ステップ実装（Phase 1→2）
  - 手動QA + 少人数テスト（開発メンバー/クローズドテスター）
  - ログ計測で改善確認

## 現状導線（実装ベース）
1. ランディング画面で `ENTER` をタップ
2. コンセプト画面で `START CONDITIONING` をタップ
3. オンボーディング開始（Welcome）
   - 新規開始 or 復元を選択
4. プロフィール入力（体重・身長・年齢・基準）
5. お気に入りビール（メイン/サブ）
6. リセット周期の選択（週次/月次/永久）
7. データ保護の注意喚起
8. スタート画面で `はじめる`
9. チュートリアル表示後、ユーザーがRecordタブへ移動し手動記録

## 課題の再整理（優先度付き）

### 優先度A（先にやる）
1. 初回記録地点までの距離が長い（LP/コンセプト/オンボ後の迷い）。
2. オンボ完了後の遷移先が曖昧で、最初の行動が決まりにくい。
3. チュートリアルが要素説明寄りで、最初のタスク完了に直結しづらい。

### 優先度B（次にやる）
1. プロフィール入力の必須範囲が初回体験に対して重い。
2. 設定選択後の自己確認（サマリー）が弱い。

### 優先度C（状況次第）
1. ビールモーダルのさらなる簡略化（現状でも一定対応済みのため）。

## 改善案（改訂版）

### 1. オンボ完了後の標準遷移を `Record` タブに統一（最優先）
- 完了直後にRecordタブへ遷移。
- 初回だけ、タブ上部に「まず1件記録してみましょう」ガイドを表示。
- CTAを2つ並列表示:
  - ビールを記録
  - 運動を記録

### 2. 分岐オンボを導入（初回入力負荷の最適化）
- 先に「最初に何を記録するか」を選ばせる。
- ビール先行ユーザーにはプロフィール入力を後回し可能に。
- 運動先行ユーザーには必要最低限のプロフィール入力を先に実施。
- ビール先行でプロフィール未設定のまま運動を記録しようとした場合は、プロフィール設定完了まで運動記録をガードする。

### 3. 設定サマリーの追加（選択可視化）
- 最終開始前に確認画面を追加。
- 表示項目:
  - プロフィール（入力済みのみ）
  - メイン/サブビール
  - リセット周期
- 各項目から直接戻って修正できる導線を配置。


### 補足) 「必須チュートリアル3ステップ目の“反映結果”はどこで見せるか？」への回答
- 結論として、**第一表示は Home のオーブ（収支表示）**を採用します。
- 理由:
  - 記録直後の変化が最も即時に伝わる主要UIであるため。
  - 初回ユーザーにとって「記録した結果がどこに反映されたか」を最短で理解しやすいため。
- 補助表示として、同ステップ内で「履歴は Cellar > Logs で確認可能」と1行案内を入れます（遷移は任意）。
- つまり、必須ツアーでは
  1. Home（何を見るか）
  2. Record（どこで記録するか）
  3. Homeオーブで反映確認（+ Logs は補助導線）
  の順に統一します。

### 補足) 「Settings・Stats・Cellarで初回アクセス時に詳細ツアーを強制発動すべきか？」への回答
- 結論として、**強制発動はしません**（煩わしさ・離脱リスクが高いため）。
- 初回アクセス時は以下の非強制導線に限定します。
- **要するに、タブごと（特に Stats / Cellar）に「初回のみの軽いヒント」を追加する想定です。**
  1. タブ内ヘルプ（`?`）に「このタブの詳細ツアー」導線を表示。
  2. 初回訪問時のみ、非モーダルの小さなヒント（チップ/トースト）を1回表示（即閉じる可・再表示なし）。
  3. `Settings` の「チュートリアル再実行」からいつでも起動可能。
- 例外として、エラー回避に直結する必須案内がある場合のみ、短い注意喚起を表示し、詳細は任意起動に留めます。

### 補足) 「データ保護の注意喚起は強制表示したほうがよいか？」への回答
- 結論として、**初回オンボ内で1回は強制表示するべき**です（データ消失リスクが高いため）。
- 推奨仕様:
  1. オンボ中に「データは端末保存」「削除時に消える可能性」を明示した必須ステップを維持。
  2. 「理解して次へ」チェック（または同等UI）を通過条件にする。
  3. 初回記録完了後、軽い再リマインド（バックアップ導線付き）を1回だけ表示。
- これにより、初回離脱を増やしにくい範囲で、重大リスクの未認知を防げます。

### 補足) 「必須ツアーでデイリーチェックが出てこない件」への回答
- ご指摘どおり、デイリーチェックを完全に外すと利用理解に偏りが出ます。
- ただし必須ツアーを長くしすぎないため、**必須では“存在だけ短く触れる”**方針にします。
- 具体案:
  - 必須3ステップの Record ステップ内で「デイリーチェックもRecordから入力できる」と1行で案内。
  - 操作方法や意義の詳細は `detailedSteps` 側へ移す（チェック項目の意味、継続メリット等）。
- これにより、初回の完走性を保ちつつ、機能の見落としを防ぎます。

### 補足) 「既存の必須チュートリアルコードは残すべきか？」への回答
- ご認識のとおり、**残して再利用する前提**にします。
- 方針は「作り直し」ではなく、既存ツアー定義を
  - `mandatorySteps`（初回必須）
  - `detailedSteps`（任意詳細）
  に分割して使い回す構成です。
- 実装上は、既存ステップ配列を完全削除せず、タグ付け/分離で移行します（重複実装を避ける）。
- これにより、Phase 1→2での変更時も文言・対象要素・順序の差分管理がしやすくなります。

### 4. チュートリアルを「必須最短 + 任意詳細」に分離
- 必須は3ステップ程度に圧縮（Home / Record / Homeオーブで反映確認）。
- カード個別説明は任意の「詳細ツアー」へ退避（初期は再編中心で、むやみに項目数を増やさない）。
- CheckStatusとLiverRankなどの説明粒度を揃える。
- デイリーチェックは必須内では存在告知のみ（1行）に留め、詳細説明は任意詳細へ分離する。
- **詳細チュートリアルの入口（想定 / 非強制）**:
  1. `?` ボタンの既存挙動（ヘルプモーダルを開く）は維持する。
  2. そのヘルプモーダル内に、"基本ヘルプ"に加えて"詳細ツアーを開始"導線を追加する（= 分岐）。
  3. `Record` / `Stats` / `Cellar` 各タブでも同様に、各タブのヘルプモーダルから該当タブの詳細ツアーを起動できるようにする。
  4. `Settings` に「チュートリアルを再実行」を設置し、必須ツアー/詳細ツアーを選択可能にする。
  5. 初回必須ツアー完了直後に「詳細ツアーを見る（任意）」CTAを1回だけ表示。
- **設計意図**: 既存のヘルプモーダル体験を壊さず、モーダル内で"読むヘルプ"と"体験型ツアー"を選べる構成にする。
- Stats / Cellar の初回タブ表示時は、詳細ツアーを自動起動せず、1回限りの軽いヒント表示に留める（Settingsは段階適用）。

## フェーズ完了時に導線がどう変わるか（Before / After）

### 現状（着手前）
`LP → コンセプト → オンボ完了 → チュートリアル（要素説明中心） → ユーザー判断でRecordへ移動`

- オンボ完了後の「次の一手」が弱く、最初の記録行動までがユーザー依存。

### Phase 1 完了後の導線
`LP → コンセプト → オンボ完了 → Recordタブ自動遷移 → 同格CTA（ビール/運動） → 初回記録`

- 変化ポイント:
  - オンボ完了直後の遷移先が Record に固定される。
  - Record上で「ビール」「運動」の2択が同格表示される。
  - チュートリアルは既存コード再利用で `mandatorySteps` 化され、最短3ステップ（反映確認はHomeオーブ）になり初回記録までの寄り道が減る。
  - データ保護注意喚起はオンボ内で必須表示され、初回記録後に1回だけ再リマインドされる。
  - 開始前サマリーにより「選択内容の確認→必要なら修正」が可能になる。

### Phase 2 完了後の導線
`LP → コンセプト → オンボ開始（最初に記録したい内容を選択） → 分岐入力 → Recordタブ → 対応CTA → 初回記録（運動前はプロフィールガード）`

- 変化ポイント:
  - オンボ冒頭で「ビール先行 / 運動先行」を選ぶ分岐が追加される。
  - ビール先行はプロフィール入力を後回しでき、運動先行のみ必要最小限入力を先に実施。
  - ただし、ビール先行でプロフィール未設定の場合は運動記録前に必ずプロフィール設定を要求する。
  - 入力負荷が一律でなくなり、目的に応じて最短化される。
  - 詳細チュートリアルは `detailedSteps` として段階的に有効化し、Home/各タブHelp/Settingsから再実行できる（Stats/Cellarの初回アクセスで自動強制はしない。Settingsは段階適用）。

### Phase 3 完了後の導線（運用安定版）
`LP → コンセプト → 分岐オンボ → Record到達 → 初回記録 → 計測に基づく継続改善`

- 変化ポイント:
  - ユーザー体験そのものは Phase 2 と大きく変えず、品質と再現性を高める段階。
  - どこで離脱しているかをイベントで可視化し、文言・順序・チュートリアル長を継続調整（必要性が確認できた箇所のみ詳細ツアー説明を追加）。
  - クローズドテスト結果と計測値をもとに、次のUI改善を安全に反映できる状態になる。

## オーナー側で確定した意思決定（1〜6）

以下は、レビュー合意により**決定事項**として固定する。

1. **オンボ必須ステップの許容長**
   - 決定: **3ステップ + データ保護必須1ステップ**を維持する。
   - 補足: 追加説明は必須側に増やさず、任意詳細へ分離する。

2. **データ保護同意UIの厳格度**
   - 決定: **「理解して次へ」チェックを必須**にする。
   - 補足: 長文既読の強制は行わず、離脱増を避ける。

3. **初回記録後リマインドの表現強度**
   - 決定: **1回のみの軽量トースト表示**（バックアップ導線付き）を採用。
   - 補足: 初回直後の強制モーダルは採用しない。

4. **分岐オンボの導入タイミング**
   - 決定: **Phase 2で導入**する（Phase 1では導線短縮を優先）。
   - 補足: 先に Record誘導・必須チュートリアル整理の効果を確認する。

5. **プロフィール未設定ガードの発火タイミング**
   - 決定: **運動記録起動時に必須ガード**を発火する。
   - 併用: 初回ビール記録後に、運動前プロフィール設定が必要である旨を軽く予告する。

6. **タブ別初回ヒントの表示対象**
   - 決定: **Stats / Cellar を優先対象**とし、初回のみ非強制ヒントを表示する。
   - 補足: Settingsは必須対象にせず、必要に応じて段階的に適用を検討する。

### 決定事項として先に固定する3項目
- A. オンボ必須ステップ数（3 + データ保護必須）
